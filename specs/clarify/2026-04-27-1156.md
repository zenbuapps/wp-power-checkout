# Clarify Session 2026-04-27 11:56 — Issue #16

## Idea

Issue #16「API KEY 要 trim 空白鍵」：管理員從第三方平台（Shopline、Amego）複製金鑰時，常不小心連同肉眼看不見的前後空白、零寬字元、全形空白一起貼到 WooCommerce 後台設定，導致儲存後 API 認證失敗、開發票失敗，且管理員與客服都很難自助發現。

需求目標：
- 後端在儲存 / 讀取設定時自動修剪所有「前後不可見字元」
- 前端在欄位失焦時靜默修剪（UX 加分）
- 中間的空白／不可見字元【不修剪】（保留可能合法的內部字元）
- 既有資料庫殘留的帶空白資料，下一次讀取時即時修剪（無感修復）
- 客服可以給出「請按一次儲存即可」的標準話術

確定的部分：
1. 後端最終防線：`ProviderUtils::update_option` 統一 trim → 所有 provider 自動受惠（含未來新增的 provider）
2. 既有資料保護：DTO `before_init()` / `after_init()` 對讀取後的字串屬性 trim → 升級無需 migration
3. 前端 UX：抽 `<TrimmedInput>` 共用元件包 `<el-input>`，在 `@blur` 事件 trim
4. 修剪字元集：半形空白、Tab、CR、LF、全形空白 (U+3000)、不換行空白 (U+00A0)、零寬空白 (U+200B)、零寬非連接 (U+200C)、零寬連接 (U+200D)、BOM (U+FEFF)
5. 範圍：所有設定欄位（包含 title / description / order_button_text 也 trim 前後，邏輯最簡）
6. 不破壞中間字元：只 trim 前後，欄位中間的空白保留
7. 修剪是靜默的，不顯示通知、不顯示警告
8. 不主動掃描資料庫批次 migrate，避免風險

## Q&A

- Q1: 自動修剪要在哪些時機觸發？
  → A: 選 A — 後端儲存時修剪（最終保證乾淨）+ 前端失焦時即時修剪（UX 加分）。三層全做（含送出前再 trim）over-engineering，兩層即可。

- Q2: 哪些欄位要被修剪？
  → A: 選 C — 全部欄位都 trim 前後不可見字元。title／description 開頭結尾的空白本來就無意義，邏輯最簡單，未來新增欄位免維護白名單。

- Q3: 除了「半形空白」之外，要不要修剪其他不可見字元？
  → A: 選 C — 涵蓋半形空白、Tab、換行、全形空白 (U+3000)、不換行空白 (U+00A0)、零寬字元 (U+200B/200C/200D)、BOM (U+FEFF)。Issue 初衷就是「肉眼看不出來」的字元都要清掉。

- Q4: 既有資料庫中已存在帶空白的金鑰，要怎麼處理？
  → A: 選 B — DTO 讀取時即時 trim（`before_init()` / `after_init()`），即使尚未重新儲存，呼叫 API 時也是乾淨的值。讀取時 trim 不會主動寫回 DB。

- Q5: 前端「失焦時即時修剪」要不要顯示提示？
  → A: 選 A — 完全靜默。Issue 風險章節明確要求「無感體驗」。

- Q6: 「中間空白」要不要做警告？
  → A: 選 A — 不做任何提示。中間空白可能合法，本次需求專注前後修剪，不擴大範圍。

- Q7: 後端修剪要在哪一層做？
  → A: 選 B + C 混合 — `ProviderUtils::update_option` 統一 trim（寫入時保護所有 provider）+ 各 SettingsDTO 的 `before_init()` / `after_init()` trim（讀取時保護既有資料，覆蓋 Q4-B）。共用 helper 例如 `StrHelper::trim_invisible($value)`。

- Q8: 前端修剪要在哪一層做？
  → A: 選 C — 抽 `<TrimmedInput>` 共用元件包 `<el-input>`。涵蓋 Q3 多種不可見字元（Vue 內建 `.trim` modifier 只 trim 半形空白，覆蓋不到全形與零寬）。

- Q9: 測試覆蓋範圍要做到哪？
  → A: 選 B — PHPUnit 整合測試（POST /settings/{id} 帶空白進去，讀回來乾淨）+ DTO 單元測試（`RedirectSettingsDTO::instance()` 從帶空白 option 讀取屬性是乾淨的，覆蓋 Q4-B）。E2E 對純資料邏輯偏重，暫不納入。
